# Frontend

React 19 + Vite 8 + TanStack ecosystem. Same-origin behind nginx
([architecture/nginx.md](architecture/nginx.md)) so cookies + CSRF are
transparent.

## Stack

| Layer | Choice | Why |
|---|---|---|
| Build | **Vite 8** | Fast HMR, native TS/JSX, modern ESM |
| Compiler | **React Compiler 1.0** | Auto-memoizes pure components/hooks — no manual `useMemo`/`useCallback`/`memo`. Wired via `@vitejs/plugin-react` v6 + `@rolldown/plugin-babel` + `reactCompilerPreset()` |
| Package manager | **pnpm 11** | Disk-efficient, fast, strict |
| Language | **TypeScript ~6** | `tsc --noEmit` on every push via hook |
| Routing | **TanStack Router** (file-based) | Typed `Link`, `useParams`, search params ([decision](decisions.md#frontend-routing--tanstack-router)). Route `loader`s preferred over `useEffect` for "fetch on arrival" |
| Data | **TanStack Query** | Caching, refetching, optimistic updates |
| Forms | **TanStack Form + Zod** | Type-safe, schema-driven, integrates with Query |
| Tables | **TanStack Table** | Headless primitives; pairs with shadcn DataTable patterns |
| State (client-only) | **Zustand** | Theme store; small, hookless, persistable |
| URL state | **(future) nuqs** | When we have shareable query params |
| UI components | **shadcn/ui + Tailwind v4** | Copy-paste, source-owned, themable |
| Icons | **lucide-react** | Tree-shakeable, MIT |
| WebAuthn helper | **`@github/webauthn-json/browser-ponyfill`** | base64url ↔ ArrayBuffer for `navigator.credentials.create/get` |
| API client | **openapi-typescript** | Reads OpenAPI → TS types; no runtime client |
| Telemetry | **@sentry/react** | DSN-driven; opt-in via env |
| Lint/format | **ESLint + Prettier** | Flat config — typescript-eslint strict, react-hooks v7, react-compiler, react-refresh, @tanstack/query, simple-import-sort, sonarjs + unicorn (cherry-picked bug rules). Mirrors the ruleset from `~/usul-policy-research-app/frontend/eslint.config.mjs` |

## TanStack Router (file-based)

The `routes/` tree drives routing. `@tanstack/router-plugin` runs at build
time and codegens `routeTree.gen.ts`. Per-route metadata via the `head`
API:

```tsx
export const Route = createFileRoute('/playlists/')({
  component: PlaylistsPage,
  head: () => ({ meta: [{ title: 'My playlists — music' }] }),
})
```

## Data fetching

Centralized query keys in [`lib/query/keys.ts`](../frontend/src/lib/query/keys.ts).
One namespace per domain (`sessionKeys`, `playlistKeys`, `roomKeys`, `mfaKeys`,
`emailKeys`), each with `all()` for broad invalidation + specific entries below. Every
entry is a function (uniform call shape):

```ts
export const playlistKeys = {
  all: () => ['playlists'] as const,
  list: (search = '') => ['playlists', 'list', search] as const,
  detail: (id: string) => ['playlists', 'detail', id] as const,
  tracks: (id: string) => ['playlists', 'detail', id, 'tracks'] as const,
}

useInfiniteQuery({
  queryKey: playlistKeys.list(search),
  queryFn: ({ pageParam }) => api<PaginatedPlaylistList>(`/catalog/playlists/?page=${pageParam}…`),
})

// Surgical invalidation:
qc.invalidateQueries({ queryKey: playlistKeys.detail(id) })
// Or namespace-wide:
qc.invalidateQueries({ queryKey: playlistKeys.all() })
```

This pattern mirrors the reference repo
(`~/usul-policy-research-app/frontend/lib/hooks/queries/query-keys.ts`).

## React conventions

Lead rule: **server state lives in TanStack Query, local UI state in `useState`,
nothing global.** There is no client store (Redux/Zustand) — the listening room +
playback are server state behind `roomKeys`, so the player reads from the cache,
not a store.

- **`useEffect` is an escape hatch for external systems only** — subscriptions
  (IntersectionObserver, ResizeObserver), `requestAnimationFrame` loops, timers
  (the search debounce), DOM/key listeners, and wiring the `<audio>` element.
  *Don't* use it for: derived values (compute them during render), mirroring props
  into state (key the component to reset instead — see `EditPanel` in
  `routes/playlists/$playlistId.tsx`), or event-driven logic (do it in the
  handler). If you're writing `useEffect` + `setState`, stop and check
  [react.dev/you-might-not-need-an-effect](https://react.dev/learn/you-might-not-need-an-effect).
- **The React Compiler is on** (`vite.config.ts` `reactCompilerPreset()`), so
  **don't hand-write `useMemo`/`useCallback`/`memo`** — it memoizes for us. Reach
  for them only when profiling shows the compiler couldn't (rare). The
  `eslint-plugin-react-compiler` rule flags code the compiler can't optimize —
  fix those rather than memoizing around them.

## Auth integration

Cookies + same-origin = transparent. The fetch wrapper:

1. `ensureCsrfCookie()` — pre-flight GET to `/_allauth/.../auth/session` if
   the `csrftoken` cookie isn't set
2. Reads cookie, sends `X-CSRFToken` header on every state-changing call
3. `credentials: 'include'` so cookies ride along

[`lib/auth/api.ts`](../frontend/src/lib/auth/api.ts) wraps the allauth
browser/v1 endpoints. Hooks in [`lib/auth/hooks.ts`](../frontend/src/lib/auth/hooks.ts)
wrap those in `useQuery`/`useMutation`.

### MFA challenge in login

When the user has MFA enrolled, allauth returns 401 with `data.flows[]`
including `mfa_authenticate`. The `isMfaChallenge(response)` helper in
`lib/auth/hooks.ts` detects this. The login route flips state and renders
the MfaChallenge sub-component (a separate file-component to avoid
re-creating the type on every render).

See [auth.md](auth.md) for the full flow.

## Theming + design tokens

OKLCH color space (perceptually uniform — easier to design consistent
shades). Tokens live in `src/index.css` as CSS variables:

```css
:root {
  --primary: oklch(0.61 0.20 264);     /* indigo */
  --accent: oklch(0.65 0.25 305);      /* violet */
  --success / --warning / --info / --destructive
  /* + sidebar tokens, chart palette, etc. */
}
```

Dark mode uses Discord-like values: `--background: oklch(0.21 0 0)`,
`--card: oklch(0.18 0 0)`. Toggle via Zustand store in
[`lib/theme/store.ts`](../frontend/src/lib/theme/store.ts), which runs
`applyInitialTheme()` **before** React mounts to prevent FOUC.

## Type generation

```
backend  →  drf-spectacular  →  /api/schema/  (OpenAPI 3 JSON)
                                      │
                                      ▼
         openapi-typescript  →  src/lib/api/types.ts
                                      │
                                      ▼
         client.ts uses the generated types in fetch wrappers
```

Run on demand:

```sh
pnpm gen:api
```

Not wired into every dev iteration — too aggressive. The contract is
explicit: backend API change → run `gen:api` → commit the regenerated
file. CI can enforce drift detection later.

## Lint, format, type-check

| Tool | Scope | Command |
|---|---|---|
| ESLint | TS/TSX | `pnpm lint` / `pnpm lint:fix` |
| Prettier | All file types | `pnpm format` / `pnpm format:check` |
| tsc | Type check (`--noEmit`) | `pnpm typecheck` |

Pre-push hook (in `.githooks/pre-push`) runs ESLint --fix + Prettier
--write + tsc on the diff against `origin/main`. Auto-fix produces
changes → the push is blocked with a "review + commit" message.

## Common patterns

- **Always use centralized query keys.** Ad-hoc string arrays drift.
- **`staleTime` of 1-2 minutes** when data has a broadcast invalidator;
  shorter when not.
- **Optimistic updates** are synchronous in `onMutate` — never `setTimeout`.
- **`useState` for ephemeral local state.** No `useState` for server data
  (it goes in Query). No `useState` for shared UI state across pages (it
  goes in Zustand).
- **No prop-drilling of server data.** Children call the hook directly;
  React Query deduplicates by query key.
- **Use route `loader`s, not `useEffect`, for "fetch on arrival."** The
  `/account/verify-email/$key` route is the canonical example —
  `useEffect`-driven mutation-on-mount has strict-mode race conditions
  and complicates the success-redirect path. The loader pattern: do
  work, populate the cache, `throw redirect()` on success, return outcome
  data for the failure renderer.

See `.claude/skills/state-management` for the full doctrine.

## Page layout

Routes under `/settings` and `/account/*` use the shared
[`SettingsPageShell`](../frontend/src/components/layout/settings-page-shell.tsx)
+ [`PageHeader`](../frontend/src/components/layout/page-header.tsx)
+ [`Breadcrumbs`](../frontend/src/components/ui/breadcrumbs.tsx) trio.
Pass `breadcrumbs`, `title`, optional `description`, optional `actions`;
the shell handles `max-w-2xl` + spacing + the breadcrumb nav. Top-level
pages (e.g. `/settings` itself) omit `breadcrumbs` — single-item trails
are confusing per NN/g UX guidance.
